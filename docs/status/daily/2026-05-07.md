# 2026-05-07 ONT Daily Log

## Executive Notes

- Started a fresh ONT context thread and reviewed the repo, documentation, and
  current public GitHub state.
- The project spine is clear: ONT is a Bitcoin-anchored payment-handle system
  with chain-derived ownership, owner-signed off-chain destination records, and
  universal public bonded auctions as the lead launch model.
- Owner-key loss should be reopened as a v1 design issue, not casually deferred.

## Decisions

- Treat universal public bonded auctions as the current lead launch direction:
  no reserved-name list, no direct-allocation lane, no pre-launch reservation
  system, and no short-name wave.
- Adopt a daily status-log discipline for ONT work. Keep daily logs in
  `docs/status/daily/YYYY-MM-DD.md` and synthesize them into group-chat updates
  near the end of each workday.
- Strong working recommendation: while a name is still backed by a live immature
  bond, the wallet that can spend the current bond UTXO should be able to rotate
  the ONT owner key by moving the bond into a valid successor bond output in the
  same transaction. This needs protocol design before it becomes a closed
  decision.
- Recovery should probably use a challenge-window model rather than instant
  owner-key rotation, so wallet compromise does not silently become immediate
  name takeover.
- Implementation decision: start with a signed recovery descriptor foundation,
  stored and served by resolvers as a distinct control-record family, before
  attempting the full wallet-proof recovery event and challenge-window state
  machine.
- Prototype recovery decision: add a distinct `RECOVER_OWNER` event rather than
  overloading `TRANSFER`; immature recovery requests must spend the current live
  bond into a valid successor bond, wait through a challenge window, and can be
  cancelled by the current owner key before finalization.
- Cleanup decision: keep the broad current project refresh as intentional
  progress, including neutral auction fixtures, browser-side auction tooling,
  parameter-review docs, private-signet script updates, and recovery descriptor
  work. Drop only generated local junk.

## Progress

- Mapped the monorepo: `packages/protocol` owns protocol/wire/signed-record
  logic, `packages/core` owns state/indexing/auction behavior,
  `packages/architect` owns transaction/artifact construction, and the CLI,
  resolver, indexer, and web apps sit on top.
- Reviewed core launch and review docs, including architecture, launch spec,
  implementation/validation, auction gap list, edge-case matrix, and review
  backlog.
- Compared local state with the latest public GitHub head. Public `main` is at
  `6ded319` (`Simplify ONT auction UX and docs`). The local dirty tree mostly
  mirrors that public refresh but still has a few terminology and smoke-script
  mismatches.
- Confirmed current code defaults through the CLI: auction window `4,320`
  blocks, soft-close window `144` blocks, maturity/lock `52,560` blocks, normal
  raise `5%`, soft-close raise `10%`.
- Added a repo-level `CODEX.md` directive for daily logging, a status-log
  template, and an owner-key recovery design note.
- Clarified what a recovery anchor could mean: immature names can use the live
  bond UTXO, while post-maturity recovery would need an optional anchor such as
  a recovery UTXO, committed recovery pubkey, original bond-spend key, or
  owner-managed recovery delegate.
- Added message-signed recovery as a possible anchor proof format. The useful
  version requires ONT to commit to a specific recovery address/script and verify
  a structured state-bound recovery message, preferably using a modern generic
  message-signing profile.
- Clarified that post-maturity recovery is not guaranteed unless the protocol
  requires, or the owner intentionally keeps, a recovery anchor after the main
  bond is released.
- Added resolver-stored recovery descriptors as a candidate design: resolvers
  can store the recovery address/script and signing profile, but authority must
  come from an on-chain commitment or owner-signed descriptor, not resolver trust
  alone.
- Clarified that recovery descriptors can reuse value-record-like signing,
  history, and resolver distribution, but should probably be a distinct
  control-record type rather than an ordinary user-facing destination value.
- Design lean: default wallet-backed recovery descriptors should reduce
  catastrophic owner-key loss, but should not replace owner keys for routine
  destination updates and transfers.
- Consensus/lean from discussion: "owner key for normal control,
  wallet-backed recovery descriptor by default" is the most elegant tradeoff so
  far compared with owner-key-only, wallet-only, permanent recovery-bond, or
  resolver-trusted recovery designs.
- Implemented the first recovery descriptor foundation: protocol signed
  descriptors, resolver storage and HTTP endpoints, CLI sign/publish/fetch
  commands, and focused tests.
- Implemented a prototype `RECOVER_OWNER` wire payload, BIP322-shaped recovery
  proof message text, pending recovery state, challenge-window finalization, and
  owner-key cancellation.
- Added recovery indexer tests for request, cancel, late cancel/finalization,
  invalid successor bond release, and checkpoint restore.
- Converted the first auction edge-case matrix items into tests for exact
  opening minimum, one-sat-under rejection, close-boundary acceptance, and
  after-close rejection.
- Cleaned local generated noise by removing ignored `.DS_Store` files and added
  `.gitattributes` so committed PDFs are treated as binary artifacts.

## Open Questions

- Owner-key recovery: should the prototype `RECOVER_OWNER` request authority
  remain bond-spend-only at consensus level, or must the indexer also verify a
  BIP322 wallet signature before entering pending recovery?
- Should bond-authorized owner-key recovery remain immature-only, or also work
  after maturity if the user intentionally keeps a recovery anchor alive?
- If recovery has a challenge window, is the current 144-block descriptor
  default enough for a real user to notice and cancel an unauthorized
  wallet-key recovery?
- Can "the original wallet can recover forever" be made compatible with normal
  Bitcoin wallet behavior, or does it require an explicit recovery key/UTXO
  profile?
- Would BIP322 support across Sparrow, Electrum, and hardware signers be good
  enough for ONT recovery, or should v1 avoid depending on message signing?
- Should post-maturity recovery be opt-in, or should ONT require every mature
  name to keep some recovery anchor?
- Should recovery descriptors be committed on-chain by hash, owner-signed and
  resolver-served off-chain, or both?
- Should the `RECOVER_OWNER` signature field remain fixed-size, or should the
  event carry a proof hash/profile while resolvers distribute variable-size
  wallet proofs?
- What exact wallet/script set should v1 support for BIP322 verification across
  Sparrow, Electrum, and hardware signers?
- How much owner-key backup burden can the product remove once default recovery
  descriptors exist?
- If both current owner key and bond-spend authority issue conflicting updates,
  what ordering and validity rules should decide the result?
- Should code defaults move toward the docs' shorter auction timing lean, or
  should the current 30-day prototype window stay until parameter review closes?
- What exact daily group-chat status format should become the public rhythm?
