## A1 — Indexer daemon enforcement (review F1) — SPLIT (per ChatLunatique)
**Problem:** `apps/indexer/src/main.ts:20-40` builds runner deps as `{blockSource, cursorStore, anchorStore}` —
no `enforcement`. `apps/indexer/src/runner.ts:70` runs `enforceBatchedClaims` only when `deps.enforcement` is set
(today only in `packages/regtest-e2e/src/enforcement-e2e.ts:222-226`). The shipped daemon serves the plain
RootAnchor read path; STATUS.md:25,99 "LE-INDEX Wired" is true only for hermetic runner injection.

**Why it is NOT a one-liner:** `EnforceBatchedClaimsDeps` (`enforce-batched-claims.ts:39-44`) needs three *live*
things the daemon has no selector for — `batchMaterial: BatchMaterialSource` (`(anchoredRoot, prevRoot) =>
BatchMaterial | null`; the served-bytes + base-leaves source — this is LE-DA-SERVE territory), `nameStateStore`,
and `policy: BatchedClaimPolicy` (launch params). Passing an `enforcement` whose `batchMaterial` returns `null`
makes **every batch "skipped"** — a silent no-op that looks fixed. So:

- **A1a — status correction (cheap, dispatch now).** Downgrade STATUS.md:25,99 from "Wired" to "runner/hermetic
  wiring only; daemon enforcement not selected." Writer lane. No code risk. **This is the only immediate cheap win.**
- **A1b — daemon enforcement selectors (real work, builder).** Build env-selected `batchMaterial` /
  `nameStateStore` / `policy` for `main.ts`, mirroring `selectIndexerStores`. **Acceptance test MUST feed real
  batch material through the daemon path and assert names are written** — not merely that `enforcement` is
  non-null. That test is the guard against the null-`batchMaterial` silent-skip. Note: the `batchMaterial` source
  ties to the LE-DA-SERVE transport, so A1b's *production* form overlaps Phase-1 DA work; a fixture source is fine
  for the acceptance test.
